# 任务规划：复杂任务先把路画出来

## 为什么需要计划

跨多个文件的任务很容易漏步骤：改了代码却没改调用方，修了问题却没运行验证，或者做到一半忘了用户原始目标。

计划的作用是保存目标、进度和下一步，不是让 Agent 写一份漂亮清单。

## 先区分任务类型

- 查询类：解释概念、读一个文件、回答一个问题，不必强行列计划。
- 执行类：要调查、修改多个文件、运行检查，适合列计划。

所有问题都先写计划，会让简单任务变慢，也会增加无用输出。

## 为什么需要三道防线

1. 用户消息层提醒复杂任务先计划。
2. System Prompt 规定计划格式和执行纪律。
3. 首次工具调用后再检查一次，拦住已经直接开干的模型。

模型可能漏掉其中一层，所以要在不同时机保护。

## 计划不能只写不做

Agent 准备停止时检查计划：

- 有未完成步骤：继续执行或说明阻塞；
- 所有步骤完成：允许结束；
- 多次提醒但状态不变：触发断路器，避免无限循环。

计划不是事实。用户改变要求或调查发现新情况时，计划要更新。

## 什么是好步骤

好步骤是可验证的动作，比如“运行类型检查并修复错误”。

坏步骤是空话，比如“处理一下问题”“继续优化”。步骤太细也不好，把每个工具调用都写成一项会让 Agent 只盯清单。

## 经验

- 计划应该帮助用户知道还差什么。
- 计划状态要写进事件日志，否则出问题时无法解释为什么停。
- 计划格式解析要宽松，`- [ ]`、编号和常见 checklist 都应尽量识别。
