# 任务完成判断：模型停下来，不代表事情做完了

## 它解决什么问题

模型可能输出“我先检查一下”，然后自然停止；工具失败后也可能只给一句很短的回答。只看模型是否停止，无法判断任务是否完成。

## 主要判断信号

- 有工具调用：通常还在工作，执行后继续。
- 调用 `task_complete`：模型明确声明完成。
- 没有完成信号但只有中间过程：注入提醒，给有限次数继续机会。
- 达到轮次、时间或 token 上限：停止并说明原因。

不要依赖正则匹配“还需要”“我来检查”来判断。自然语言太多，误判会干扰模型。

## Goal 模式

`/goal` 是外层循环：做一轮、运行验证、把失败输出反馈给下一轮，直到完成、阻塞、超时或预算耗尽。

普通 Agent Loop 只负责一轮；Goal Loop 负责多轮编排、验证和总预算，分开后更容易排查。

## 验证命令为什么不是绝对权威

验证命令有用，但会受操作系统和环境影响。Unix 重定向语法放到 Windows 可能直接失败，命令失败不一定代表代码没完成。

因此验证结果主要用于反馈和日志。模型已经明确完成时，验证失败记录 warning，但不能让系统因为环境命令写法进入死循环。

## Stop Hooks

停止边缘可以挂检查器：

- 安全检查优先，发现危险操作就停止；
- 完成检查拦截过短、没有结论的回答；
- 第一个明确结果生效，避免一个检查说继续、另一个说停止。

## 必须有的保护

- 最大轮数和最大时间；
- 连续相同验证输出检测；
- 用户停止绕过普通运行锁，立即生效；
- 所有结束原因写入日志；
- 验证阶段异常不能让 Worker 直接崩溃。
