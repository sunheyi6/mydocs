# Agent 循环：模型不是只请求一次

## 它解决什么问题

普通聊天是“用户问 → 模型答”。编码 Agent 不够：模型必须先读代码，再修改，再跑检查，然后根据结果决定是否继续。

所以 Agent 的核心不是一次请求，而是一个受控循环：

`请求模型 → 有工具就执行 → 把结果放回上下文 → 再请求模型`

## 一轮里发生什么

1. 准备当前消息、项目规则和相关记忆。
2. 请求模型并接收流式输出。
3. 如果有工具调用，校验参数并执行。
4. 保存模型输出和工具结果。
5. 根据结果判断继续、等待用户还是结束。

模型每次只能看到已经发生的事实。工具没有返回的内容，模型不能靠猜。

## 为什么要限制循环

模型可能重复调用同一个工具，也可能因为验证命令一直失败而不停重试。循环必须有：

- 最大轮数；
- 超时；
- token 预算；
- 连续相同结果检测；
- 用户停止信号；
- 清楚的结束分类。

限制的目的不是尽快停，而是防止 Agent 在没有新信息时继续消耗资源。

## 普通 Run 和 Goal Loop

普通 Run 解决一次用户请求，内部按轮执行。

Goal Loop 在外面再套一层，负责“做一轮 → 验证 → 把失败反馈给下一轮”，适合长任务。两者要分开：内层关注一次运行，外层关注总预算和验证。

## 状态要和 UI、日志共用

每轮开始、工具开始、工具结束、回合结束和 run 结束都应有事件。UI 用它显示进度，日志用它排障，不能让两套状态各自猜。

## 常见错误

- 只看模型有没有输出，不看工具结果。
- 工具失败后原样重试。
- 没有最大轮数，靠模型自觉停止。
- 用户停止被普通 session 锁挡住。
- 当前会话切换后，旧 run 的事件写进新会话。

相关实现通常分成 `agent-loop`、`goal-loop`、`turn-decision` 和 `stop-hooks`。
