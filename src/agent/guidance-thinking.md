# 运行中引导：用户中途补充要求，不能把进度丢掉

## 它解决什么问题

用户可能在 Agent 工作时说“后面用中文”“不要改这个文件”“结果分两点写”。等任务结束太晚，直接中断又会丢掉已经执行的工具结果。

## 当前思路

每个 session 保存一个引导队列：

1. 用户发引导时只入队，不 abort。
2. 下一轮模型请求前取出队列。
3. 把引导作为新的 user message 加入上下文。
4. 写入历史和运行事件。

这样模型看到的是完整旧事实加上新要求，已发生的事情不会被改写。

## 为什么有两个注入点

- 每轮开始前：处理上一轮执行期间收到的引导。
- Stop 边缘：处理用户在最后一轮流式输出期间发来的引导，必要时多跑一轮。

第二个点防止“引导刚到，模型已经结束”的竞态，但仍受轮次和时间预算限制。

## 为什么绕过普通 session 锁

普通 prompt 要串行，避免两个任务互相覆盖；引导只是向队列追加数据，不应该等待当前 prompt 结束。它不改 abort controller，也不改已经输出的内容，所以可以走快速通道。

## 边界

- 引导不能改变已经完成的文件操作。
- 空引导丢弃，多条引导按顺序处理。
- 引导要立即落盘，运行崩溃后不能出现历史缺失。
- 引导不是权限绕过，仍受工具权限和安全检查限制。
