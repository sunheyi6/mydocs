# 流式输出：先让用户知道 Agent 没卡住

## 为什么要流式

Agent 可能运行很久。等所有工作结束才显示，用户无法判断是在执行、等待还是死循环。流式输出让用户看到当前进度，也能及时停止或追加引导。

## 流式不是只传文字

实际事件至少包括：

- thinking/text 的增量；
- 工具调用开始、参数和结束；
- 工具运行状态；
- 回合开始和结束；
- token 和整体状态。

这些事件要带上 sessionId，多个会话并行时不能串到一起。

## 为什么要保存 blocks

按到达顺序把 thinking、text、tool call 保存成 blocks。相同类型的增量合并，类型变化新建块，工具按调用 ID 跨轮合并。

这样 UI 能恢复真实时间线，完成后也能重新渲染；下一轮空的 toolCalls 不会覆盖前面已经记录的工具。

## 流式中不要急着说最终回答

当前文字块只是最后到达的内容，后面仍可能调用工具。只有流结束后才能确认最终回答。中间文字放过程区，最终 text 放主回复区。

## 用户停止和会话切换

停止要中断当前流并清理后台进程；切换会话前先中止旧 run，避免事件写进新 session。停止、abort 等控制信号不能被普通消息锁长时间挡住。
