# 会话：切换要快，历史要完整

## 为什么切换会卡

一个会话可能有很多轮消息、工具输出和思考内容。切换时一次性读取和渲染全部内容，会造成等待；快速连续切换还会出现旧请求覆盖新会话。

## 先展示最近内容

切换目标会话时先加载最近约 10 条消息，立即展示；完整历史在后台读取并写入缓存。用户先看到上下文重点，后面再补齐细节。

这只是 Renderer 的展示优化，Worker 仍要拿完整历史，不能因为 UI 只显示 10 条就丢 Agent 上下文。

## 防竞态

每次加载都有请求序号。只有仍然对应当前会话的请求，才能更新界面；旧请求最多更新缓存，不能覆盖用户当前正在看的会话。

切换会话前要中止旧 run，避免旧流式事件继续写入新会话。

## 标题为什么用规则先做

会话标题的作用是导航，不是深度总结。先取第一条用户消息，清掉 Markdown 和多余标点，限制长度，速度快、成本低，也不会影响消息保存。

以后接 AI 标题时也应先保存默认标题，再异步更新，不能让标题生成阻塞会话。
