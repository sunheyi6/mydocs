# 工具操作展示：让用户能核对 Agent 做了什么

## 为什么需要它

“修改完成”太模糊。用户需要看到文件、增删行数、命令、工作目录、输出和退出码，才能判断结果是否可信。

## 为什么工具层返回结构化数据

UI 如果解析人类可读的 output，文案一改就会坏；用整个 Git diff 推断命令造成的变化，又会把用户原有改动算到 Agent 头上。

所以：

- 文件工具返回路径、状态、增删行数和错误；
- bash 返回命令、cwd、stdout、stderr、exitCode 和 signal。

Renderer 只展示这些字段，不猜测间接修改。

## 展示原则

- 文件操作默认展开，方便确认变化。
- 命令操作默认折叠，避免输出占满聊天。
- 开始时显示运行中，结束后更新成功或失败。
- 历史消息保存参数和结果，重新打开仍能还原卡片。
- 旧消息缺少结构化结果时只显示已有信息，不伪造统计。

## 核心经验

事实在产生它的那一层计算：行数由文件工具算，退出码由命令工具提供，UI 只负责呈现。
