# Edit 工具：按代码片段改，不要迷信行号

## 它解决什么问题

行号会随着前面的修改变化，模型也不擅长数行。edit 使用 `oldText → newText`，先找到旧代码片段，再替换成新片段。

## 修改流程

1. 校验路径在工作目录内。
2. 读取文件，处理 BOM 和换行符。
3. 先精确匹配，失败后才做有限模糊匹配。
4. 确认匹配唯一，检查多个 edits 没有重叠。
5. 从后往前应用修改，避免前面索引被改变。
6. 恢复原换行和 BOM，再写回。

## 为什么需要模糊匹配

Windows 的 `\r\n`、BOM、智能引号、全角空格和行尾空格，都会让看起来一样的文本匹配失败。

模糊匹配只做有限归一化。它是兜底，不是让工具随便猜位置；片段不唯一就报错，要求 Agent 重新 read。

## 为什么同一文件要排队

同一轮可能并行执行 write 和 edit。两个工具同时读写同一文件会产生竞态。文件 mutation queue 让同一物理文件串行，不同文件仍可并行。

## 错误要能推动下一步

- 找不到：提示重新 read。
- 多处匹配：要求缩小片段。
- edits 重叠：要求合并。
- 没有变化：返回 no-op，不假装成功。

## Agent 使用经验

修改前先 read；oldText 选足够独特的上下文；局部改用 edit，整体重写用 write；edit 失败后不要原样重试。
