# Grep 工具：先找到相关代码，再决定读多少

## 它解决什么问题

直接读取整个仓库太慢，也浪费上下文。Agent 更常需要找定义、调用点和错误字符串，grep 负责先定位。

## 为什么用 ripgrep JSON

纯文本遇到 Windows 路径、空格和上下文行时不稳定。`rg --json` 给出路径、行号和匹配文本，程序不需要猜分隔符。

## 上下文为什么自己读

`rg --json` 的上下文输出不够稳定。先拿匹配位置，再自己读取文件行，并缓存同一文件的行内容，格式和行号都更可控。

## 为什么达到 limit 就停止

宽泛关键词可能命中几万行。达到 limit 后提前 kill ripgrep，省 CPU 和时间，并告诉 Agent 缩小 pattern、path 或 glob。

## 输出边界

- 单行过长要截断并提示；
- 总输出约 50KB 后停止继续拼接；
- 默认匹配数约 100；
- 需要全文时使用 read。

截断必须明确，否则 Agent 会误以为结果完整。

## Agent 使用经验

不知道文件在哪：先 glob，再 grep，再 read。结果太多时缩小搜索范围，不要只把 limit 调到很大。搜索结果提示不完整时，要继续取证。
