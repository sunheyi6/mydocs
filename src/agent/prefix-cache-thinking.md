# 模型前缀缓存：它不在你这里，你要去命中它

## 缓存在 provider，不在 Agent

大模型的前缀缓存（KV cache / prompt cache）由 provider 侧维护。Agent 自己不存 KV，能做的是：把每次发出去的请求组织成一个“字节级稳定的前缀”，让 provider 的自动前缀缓存尽量命中，并在没命中时说得清原因。

不要把这件事和 Agent 本地缓存混在一起。本地缓存复用自己的计算；前缀缓存是命中别人已算好的结果。

## 请求结构：固定段在上，动态段在下

把请求看成有序段，从上到下：

`工具 schema → system prompt → 对话历史 → provider 选项`

provider 按前缀逐 token 匹配缓存，前缀越长越稳定，命中越多。所以**越靠上越要稳，越靠下越允许变**。对话历史天然是“尾部追加”，和前缀缓存最合拍；工具 schema 在最上面，反而最需要花力气保稳。

每段算一个内容 hash，并标记是否可缓存（provider 选项通常不算）。

### 为什么把工具 schema 排在 system 之前

这个顺序有两层含义，不要混在一起：

- **它是诊断/度量用的“概念顺序”**，用来把请求分成可缓存段、算每段 hash、定位第一个变化段。provider 实际拼装顺序由 provider 自己决定，不同 provider 不同：有的把 tools 和 system 当成并列的顶层字段（各自独立打缓存断点），有的把 tools 排在 system 之前拼装。命中取决于 provider 实际字节，不取决于这里的诊断顺序。
- **但把工具放最前是个取舍**：原则上“更稳的段放前面，前缀才能更长”。可现实是 system 和 tools 都会变——system 要承载编排模式、运行环境，常常动态拼接；tools 因为懒加载、权限 gating，可见集也会变。所以这个顺序是在表达“我打算把哪段打造成最稳的基底”。

把工具 schema 放最前的实际收益在诊断侧：如果 system 排在前面，system 一变就会报“从 system 段开始失效”，但实际工具那段缓存可能还能命中，诊断就粗了。工具在前，诊断先确认工具段稳不稳，失效点能精确定位到 system 或历史段。

一句话：**顺序不决定 provider 怎么缓存，顺序决定失效诊断从哪里开始切、前缀从哪里开始断。** 把你打算保稳的段放前面。

### 落到自己的项目

- 如果你的工具描述完全写死、永不改，那它就是最稳的段，放最前最划算；system 做 AB test 的变化只影响 system 段之后的缓存。
- 如果你的 system 也会动态拼接（带时间、带用户态、带运行环境），把“会变的那部分”挪到尾部 user message，别塞进 system，否则 system 段每次都失效，连累它之后写入的缓存也白写。
- 真正决定命中率的是“前缀字节是否稳定”，不是“谁排在第一个字段”。顺序只是帮你把失效点切到一个更省的位置。

## 两个指纹：可缓存前缀 和 全量

- **可缓存前缀指纹** = 工具 + system + 选项，不含对话历史。这是“provider 能缓存的那段”。
- **全量指纹** = 再加历史形状。

每轮请求都拿当前前缀指纹和上一轮比，分类出原因：

`首次 | 模型或 provider 变了 | system 变了 | 工具 schema 变了 | 选项变了 | 稳定`

`稳定` 就是这一轮可缓存前缀和上一轮字节一致，provider 应当命中。任何非稳定原因都要上报，这样在命中 token 数之外还能解释“为什么这次没命中”。

## 段级失效诊断：定位第一个变化的段

只说“没命中”不够。逐段比对 hash，找到**第一个发生变化的可缓存段**：是工具 schema 第 2 个变了，还是 system 变了，还是第 N 条消息变了。等于把缓存失效点精确到段。排障时最常问的就是“缓存从哪里开始断的”，这个诊断直接回答它。

## 工具 schema 在最上面却会变，怎么办

工具 schema 排在最前，按理最该稳定，但每步可见的工具集会变（工具组懒加载、权限变化、经济性 gating）。可见集一变，schema hash 就变，前缀从头失效。

让最前段也尽量稳的办法：

- **全量注册表和可见子集分离**。全量表按工具名排序后固定；只把当前可见子集发上线。缓存指纹只算可见子集，这样“一个还没加载的工具 schema 变了”不会误报成前缀变了。
- **工具按名排序固定顺序**，避免注册顺序抖动让 schema 字节漂移。
- **同一轮内可见集只增不减**。某个工具组这一轮加载后，后续步只扩张不收缩，减少跨步的前缀抖动。
- 真的变了就如实报 `工具 schema 变了`，不要假装命中。

## 历史在下面：追加，不改写中段

历史段每轮往后追加，不动前面——这是前缀缓存最喜欢的模式。但压缩和工具结果裁剪会动到历史中段，要约束：

- 这些操作**只动历史**，不碰可缓存前缀（工具+system+选项），所以前缀那段缓存照常命中，只是历史那段从变化点之后重算。
- 工具结果裁剪要让“原文和占位符共享同一个内容指纹”，使裁剪成为表示变化而非历史分叉，已接受的压缩投影不被推翻。否则反复改写历史中段会让缓存反复失效。

## 本轮上下文单独放尾部

每轮变化的临时上下文（运行环境、沙箱状态、本轮提醒）不要塞进 system prompt。把它和当前用户消息拼在一起，作为最后一条消息追加。这样会变的东西只影响尾部消息，稳定的 system 段缓存继续命中。

## TTL 与中断：闲置本身就是未命中

前缀稳定不代表一定能命中。KV cache 有寿命，provider 默认保留时间可能短得离谱——有的只有几分钟。一次编译、一轮测试、一杯咖啡、停下来看 diff，都可能超过保留窗口。

从用户看，会话是连续活的；从 provider 看，它只看到一串孤立请求，中间没有流量。窗口一过，存好的 KV 就没了。下一轮哪怕 prompt 一个字都没变，整段前缀也要按未缓存价重新算、再重新写回缓存。这就是为什么一句 "continue" 在缓存过期后会异常贵。

应对：

- 不要假设 provider 会无限期保留。长任务里如果要在模型请求之间插入耗时操作（构建、测试），要意识到这段空档在吃掉缓存。
- 部分 provider 暴露更长保留选项，可以请求长 TTL，但这只是请求，挡不住内存压力下的驱逐，也不保证跨请求一直存活。
- 真正能控的是前缀稳定性；TTL 是 provider 侧事实，Agent 只能记录它、在它过期时如实报 "闲置超时"。

## 未命中的经济代价

缓存读通常有折扣，因为贵的 prefill 已经做过了；缓存写可能有溢价，因为 provider 要替你留存状态。命中便宜，未命中贵，差距来自整段前缀被重新 prefill。

长会话里，重读旧输入的成本可能远超生成下一句回答。所以一个看似无害的短请求，在缓存失效后账单会跳一截。

更隐蔽的是激励错位：用户想要高命中（省延迟省钱），自持 GPU 的 provider 也想要（同样硬件跑更多请求），两边利益一致。但中间的网关或转售商如果按未缓存 token 计费收入，miss 反而让它赚更多。它未必会主动破坏缓存，但意味着缓存表现必须可观测——不能只从意外变大的账单反推哪里出了问题。

## 未命中常见原因

把诊断维度和这几条放一起，方便排障时对照：

1. 闲置超 TTL。构建、测试、看 diff、开会、午休超过了 provider 保留窗口。
2. 模型或 provider 切换。KV 是模型专属，一般不跨 provider 迁移。
3. 会话分支。rewind、fork、另一条分支即使 session ID 不变，token 序列也只部分重叠；可能只保住最开头的 system 段。
4. 压缩或手动改写历史。这是故意的表示替换，会建立新前缀。
5. 工具或推理级别变更。增删改工具定义动的是请求靠前段，除非模型支持消息锚定的加法式加载且改动纯加法。推理级别变更通常同理。
6. 动态 system prompt。时间戳、随机值、变化的运行环境塞进 system，会让它之后的缓存全白写。
7. 扩展对消息或 payload 的变换。Pi 侧 transcript 稳定，到 wire 上被扩展改写，等于前缀不稳。
8. provider 路由与驱逐。prompt 一字不差也可能 miss，因为 KV 块没落在请求被路由到的那台 GPU 上。

## 度量：把 provider 的命中数接上前缀诊断

provider 回来的 usage 字段方言很多（缓存读、缓存写、命中、未命中），要统一归一化。然后把命中 token 数和前缀指纹、失效原因关联，于是每次请求都能回答：

- 这次命中了多少、写入了多少；
- 没命中是因为前缀哪一段变了；
- 命中是稳定续上，还是换了段重新起算。

## 经验

- Agent 不维护 KV，只维护请求前缀的字节稳定性。
- 越靠上的段越要稳；工具 schema 在最上面，反而最需要保稳。
- 全量集和可见集分离，缓存指纹只算可见集。
- 压缩和裁剪只动历史表示，不动 lineage，避免反复推翻前缀。
- 没命中不可耻，假装命中或说不清为什么没命中才危险。
