# 上下文压缩：对话太长时，保住任务重点

## 为什么需要它

长任务会积累大量工具输出和中间文字。上下文太长会导致请求失败、延迟变高、费用增加，还可能让模型看不到当前真正重要的问题。

压缩不是简单删除旧消息，而是把历史变成一份还能继续工作的状态说明。

## 两种压缩

### 容量保护

快到模型上限时触发，目标是“下一次请求不要超限”。要预留 System Prompt、工具结果和模型输出的空间，不能等到完全满了才处理。

### 语义压缩

内容还没超限，但已经被重复的命令和中间话淹没时触发。它重新整理目标、已完成工作、关键文件、错误、约束和下一步。

还有一种更轻量的处理只针对工具结果：把过大的单条结果替换成占位符，不动语义，见下文“工具结果裁剪”。

## 摘要必须回答什么

- 用户最终要什么；
- 已经做完什么；
- 改了哪些文件；
- 哪些检查成功或失败；
- 还剩什么；
- 有哪些约束；
- 下一步应该做什么。

只写“前面完成了很多工作”没有用，因为模型无法据此继续。

## 工具结果裁剪：替换，不是浓缩

工具输出往往是上下文里最占空间的部分。不必等到整段历史需要压缩，可以在工具结果刚产生、或下一轮请求前，把过大的单条结果就地替换成一个占位符，原文落盘归档。

它和语义压缩不同：语义压缩会重新整理和浓缩历史；工具结果裁剪不动语义，只换表示——把一条大结果换成一张“此处有归档”的元数据卡片。模型看到的不是浓缩版，而是“这里有一个被归档的大结果，指纹是 xxx，恢复路径是 yyy”。

保证语义不变靠三件事：

- **占位符自描述**：带工具名、调用 id、原始大小、内容指纹、归档位置。模型知道这里曾有一个工具结果，而不是看到一个无解释的空缺。
- **原文与占位符共享同一个内容指纹**：这样裁剪被当成“表示变化”而非“历史分叉”。已接受的语义摘要不会因为裁剪被推翻；而任何真正的内容篡改仍能被检出。
- **还原由 runtime 主动做，且校验指纹**：重放上下文时按预算和检索结果决定把哪些归档喂回去，读回后核对指纹，对不上就丢弃。落盘失败时保留原文，绝不出现“内容丢了但模型以为有”。

它和语义压缩分工叠加：占位符替换省单条结果的体积、不动语义；语义压缩浓缩多轮历史。先做便宜的裁剪，把昂贵的语义压缩留作最后手段。

## 为什么要保护缓存

可以把请求看成三步：

- A：用原上下文请求主模型；
- B：用相同前缀请求轻量模型生成状态摘要；
- C：在原前缀后追加摘要，再请求主模型。

这样压缩请求可以复用前缀缓存，避免每次都重新计算整段历史。

## 什么时候不要压缩

- 工具还在运行，结果没有回来；
- 当前轮还有未处理的工具调用；
- 摘要模型返回空内容或格式明显错误；
- 只有很少历史，压缩成本不划算；
- 用户刚插入的引导还没有写入上下文。

压缩失败时不要用半截摘要覆盖原历史。保留原文、记录原因，必要时安全停止。

## 实际经验

- 工具输出通常比模型文字更占空间，先限制工具输出。
- 摘要里放文件名、符号名和结论，不要塞大段代码。
- 摘要也需要校验和日志：压缩前后 token、耗时、触发原因和是否成功。
- 压缩是降级能力，不能成为 Worker 崩溃的新原因。
