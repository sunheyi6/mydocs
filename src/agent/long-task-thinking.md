# 长任务不丢注意力：把任务状态搬出模型，每次重新投喂

## 它解决什么问题

任务长于一轮、长于一次 Run，甚至长于一个进程的存活时间时，模型会丢目标。原因不是模型变笨了，而是目标只活在模型上下文里：上下文一压缩、进程一重启、上一轮一结束，目标就没了来源。

常见表现：

- 压缩历史后，模型忘了用户最初的约束；
- 跨轮续跑时，模型把上一个长任务当成全新对话重新开始；
- 进程重启后，内存里的循环计数、当前 todo、工作目录归属全都消失；
- 模型声称“做完了”，其实只完成了自己记得的一小部分。

根本症结：**把任务状态交给模型上下文，等于把真相交给一个会忘的东西。**

## 核心思路：给任务一个比 Run 更长的身份

普通 Run 回答“这次执行怎么开始和结束”。长任务还需要回答另一组问题：

- 这个长期目标的稳定 ID 是什么？
- 已经尝试过几次？预算还剩多少？
- 当前是运行中、等批准、还是被预算停住？
- 下一次执行该继承哪些状态？
- 工作目录是不是上一次离开时的那个？
- 谁有权接管下一步？

解决方法是给任务一层比 Turn/Run 更长的 envelope：一个长程目标从开始到终止共享一个稳定 ID，跨越多个有预算边界的执行切片（Attempt）。跨切片的事实写进 append-only 的任务日志，下一步状态从日志投影出来。

一句话：

> A task is longer than a turn; durability begins by giving the task its own log.

## 不靠压缩记目标，靠每个切片重新投喂

很多人直觉是“靠上下文压缩保住目标”。这不够：

- 压缩是有损投影，rolling 摘要会积累误差；
- 压缩只回答“过去发生了什么”，不回答“接下来从哪继续、预算还剩多少、当前 todo 状态”；
- 压缩出的目标不是权威，模型在摘要里写“测试已通过”仍只是概括，不能升级成事实。

压缩解决的是**上下文窗口要爆**的问题，是辅助层。真正防丢注意力的是另一条线：

- 任务日志 append-only 保存 inventory（面对的工作面）、todos（下一步具体做什么）、有界证据（最近观察到了什么）、自检状态；
- 每次新切片启动时，从日志投影出最新的、有界的、结构化的任务状态，**主动塞回 prompt**；
- 模型不需要从压缩历史里去猜目标，而是直接拿到一份“你在哪、该做什么”。

这就是主线：**把任务状态从模型脑子里搬到 Runtime 的 append-only log 里，每个切片重新投喂一次。模型可以忘，log 不会忘，下一切片又会被重新告知。**

## 四层身份，不要混成一层

| 身份 | 回答的问题 | 典型生命周期 |
|---|---|---|
| 长程任务（TaskRun） | 这个目标从开始到终止是谁？ | 跨切片、可跨多次执行 |
| 执行切片（Attempt） | 这次有预算边界的推进是谁？ | 一次 task-level try |
| 回合（Turn） | 这次送给模型的输入和回复是哪一轮？ | 一次交互 |
| 运行（Run/Invocation） | 这次模型与工具循环如何开始和结束？ | 一次 Runtime execution |

长程任务的进度等于：一串有界切片 + 切片之间的持久任务事实 + 明确选择的续跑状态。**不是**一个永不结束的单一 Invocation。

## 为什么要给循环一个可证明的边界

无人值守循环如果只有“继续直到完成”，就没有安全的终止条件。至少三层硬上限：

- 最多启动多少个切片；
- 所有切片累计的工具步数；
- 切片循环的总墙钟时间。

自定义决策策略可以请求 continue/retry，但硬 cap 必须重新生效。每个回环都要过切片边界和预算判断；等待批准恢复时同样开新切片。

没有硬 cap，循环不会“一直记得任务”，而是会在跑偏后无限消耗。

## 续跑不是热恢复，是新切片

“继续任务”至少有三种不同含义，不能混成一个 resume：

1. **同一工作区内的新 Turn**：上一轮因步数上限或不完整工具调用结束，用固定续跑 prompt 再发一轮，受 Turn 数和步数限制。
2. **新切片**：重跑续跑指令，任务状态从日志投影进新 instruction；旧切片的 Runtime 事件可选地显式拼进下次上下文。
3. **暂停权限的恢复**：任务因需要人工批准而 parked，批准后开一个新切片，把授权事实写进日志。

共同点：**续跑单元都是新的 Turn/切片，不是把旧调用栈复活。** 一个已经结束的模型流、已经退出的工具进程，不会被“指令指针恢复”。恢复的是任务事实，不是执行现场。

## 进度是跨切片的有界投影，不是上轮回复

长任务不能只把上一切片的自然语言回复塞回 prompt。要用结构化、append-only 的进度事实：

- **Inventory**：当前面对的工作面是什么——文件/产物、状态、用途、开放问题。它是 latest-state snapshot，新的覆盖旧的，旧 snapshot 留在日志里。
- **Todos**：下一步具体做什么——每项带 ID、优先级、状态、内容。投影能稳定找到当前 in_progress 项，也能保留历史变化。
- **有界证据**：最近观察到了什么——只保存有限摘要、截断引用、来源链接、改动元数据，不把大 stdout、文件正文、原始 diff 全量塞进 prompt。
- **自检状态**：模型的公开自检——公开理由、命令/产物证据、执行卫生。它是 advisory 任务状态，不是隐藏权威。

下一切片启动时从这些投影生成有界文本：只展示部分 inventory/todos、最近证据和有限条目。**这是任务日志的投影，不是完整 replay，也不声称省略的原始内容仍在 prompt 里。**

## 自检是反馈面，不是信任面

模型做完复杂迁移后回一句“应该可以正常工作”几乎没有工程价值。要让它能检查自己，又不让“我检查过了”变成自我授权：

1. **先声明检查边界**：检查哪些最终产物、临时输出放哪、准备看哪些路径——没有事先 plan，事后无法区分计划内产物和意外副作用。
2. **提交公开证据**：实际执行了什么命令、exit code、产物存在性，带来源引用。
3. **来源闸门**：检查内容只能引用公开任务材料，拒绝混入隐藏测试、私有阈值等不该进反馈的信息。
4. **卫生审计**：检查有没有越出临时目录、产生未计划的新文件。
5. **有界修复**：检查不通过时，给一次额外修复轮，不是无限自循环。

语义完成需要多项条件同时成立：有 accepted 自检、status=pass、有沙箱证据、workspace guard 已执行、没有未清理的副作用、plan 与证据一致、todos 全部完成。最终状态仍标 `advisory: true`——**Self-check can produce evidence and feedback; it cannot manufacture trust.**

## 不要把这套机制误解成什么

- **不是一个无限大的 Turn**。Turn 仍要有清晰边界，长任务靠多个有界切片前进，而不是把一个模型流永远开着。
- **不是把完整历史反复塞回模型**。跨切片状态来自有界进度投影，原始事件只在策略显式选择时才重放。
- **不是 Memory 系统**。这套状态服务一个具体长期目标，跨任务的长期偏好和知识属于另一套生命周期和治理。
- **不是工作目录本身**。任务日志可以引用工作目录 lease 或 snapshot，但不能替代文件 bytes、外部容器或远程 volume。
- **不是同一个 Invocation 的热恢复**。续跑和恢复单元都是新 Turn/切片，恢复 provider stream 或 live 工具进程是更强也更脆弱的协议，不作为默认。

## durable 必须说得精确

持久化的是**控制历史和投影来源**，不是整个执行世界：

- 持久：任务 ID、已 append 的任务事件、事件中记录的切片/进度/决策/权限/生命周期、指向运行和产物的引用。
- 不持久：临时工作目录的所有当前 bytes、正在运行的模型流、工具进程的 live handle、内存中的循环闭包、外部 executor 是否仍持有同一工作目录。

所以要区分两句话：

- “任务状态可以回放重建” ✓
- “进程重启后执行会自动接上” ✌（还需要 scheduler claim、工作目录 snapshot、owner fencing、预算可恢复会计）

真正跨进程可接管的恢复，需要把任务 high water、运行 high water、工作目录 snapshot、预算会计、owner fencing 绑成同一个 checkpoint 协议。

## 必须保护的不变量

1. 任务有稳定身份：同一长程目标的所有切片共享一个 ID。
2. 每次执行有独立切片边界：不靠进程内隐式循环。
3. 先 append 再 project：任务状态来自已 append 事件，不从可变投影反向写真相。
4. 续跑必须留痕：continue/retry/park/resume 都要有事件，不靠隐式循环。
5. 硬 cap 优先于策略：自定义决策不能绕过硬上限。
6. 事后授权无效：迟到的批准不能改写旧工具调用当时的权限事实。
7. 进度有界且带来源：后续 prompt 只接收有限投影，并保留任务/切片/来源引用。
8. 工作目录归属显式：没有 snapshot/lease 证据时，不得声称跨切片工作目录连续。
9. 运行事实独立：任务事件引用运行执行，不复制或改写运行事件真相。
10. 冲突可观察：损坏行、ID 不匹配、多个终止事件都要产生 warning。

## 经验

- 把任务状态交给模型上下文 = 把真相交给一个会忘的东西。状态要落进 append-only log。
- 压缩是辅助，续跑投喂是主线。压缩让历史不爆，投喂让目标不丢，二者不可互换。
- 每个新切片都从一个干净、有界、最新的任务状态视图开始，而不是从可能漂移的压缩历史里猜目标。
- 循环必须有硬 cap，否则“一直记得任务”会退化成“跑偏后无限消耗”。
- 续跑是新切片，不是旧调用栈复活。恢复的是任务事实，不是执行现场。
- durable 要说精确：持久的是控制历史，不是整个执行世界。
