# 提示词：把 Agent 的工作边界说清楚

## 它解决什么问题

模型不知道项目目录、权限和工具规则。只说“帮我改代码”，它可能没读文件就猜、工具参数写错、做一半就停，或者把中间话当成最终答案。

系统提示词的作用是给模型一套稳定的工作方式，不是用长篇文字替代程序保护。

## 推荐顺序

1. 角色：你是软件工程 Agent。
2. 能力：可以读、搜、改文件和跑命令。
3. 行为：先了解，再行动；改完要验证。
4. 工具纪律：工具怎么选、失败后怎么办、什么时候停止。
5. 环境：工作目录、系统、语言和权限。
6. 项目上下文：规则、Skills、相关记忆和教训。
7. 结束要求：只在完成后给最终回答，并给出结果。

顺序的原因是先确定身份和原则，再放容易变长的动态内容。

## 规则要写得能执行

- 不要写“认真一点”，要写“修改前先 read”。
- 不要只说“工具用于读文件”，要说“需要了解现有代码时使用”。
- 错误信息要让模型知道下一步，比如“路径不存在，请先用 glob 搜索”。
- 规则少而关键，不能把提示词写成手册。

## 为什么要重复重要规则

用户消息、System Prompt 和首轮检查出现在不同时间。复杂任务先计划、完成后调用 `task_complete`、不要重复失败工具等规则，可以在不同层各有一次保护。

这不是靠重复文字解决问题，而是降低模型漏看某层指令的概率。安全和完成判断仍要放到代码里。

## 工具描述如何省 token

提示词只放一行式工具摘要，告诉模型用途和关键限制；完整 Schema 交给 API 层。项目规则、记忆和教训也应按需加载，不能把全部内容塞进每次请求。

## 常见错误

- 规则互相矛盾。
- 工具说明长到模型忽略真正参数。
- 把所有历史和教训写进 System Prompt。
- 试图只靠提示词阻止危险命令。
- 没有明确“什么叫完成”。
